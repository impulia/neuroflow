# Publishing Guide

This document outlines the automated process for releasing and publishing new versions of Neflo.

## Automated Versioning and Releases

Neflo uses [release-plz](https://github.com/MarcoIeni/release-plz) to automate the release process. Versioning is based on [Semantic Versioning (SemVer)](https://semver.org/) and is inferred from [Conventional Commits](https://www.conventionalcommits.org/).

### How it works

1.  **Merge to Main**: When a Pull Request is merged into the `main` branch, the `release-plz` GitHub Action is triggered.
2.  **Release PR**: `release-plz` analyzes commit messages since the last release and opens a Release PR that bumps the version in `Cargo.toml` and updates `CHANGELOG.md`.
3.  **Tag Creation**: When the Release PR is merged, `release-plz` creates a new Git tag (e.g., `v0.3.0`).
4.  **Build & Release**: The tag push triggers the `Release` workflow, which builds macOS binaries for both `aarch64` and `x86_64` using `cargo-dist`, then creates a GitHub Release with the compiled archives attached.
5.  **Self-Update**: Users can run `neflo self-update` to download the latest release binary directly from GitHub.
6.  **Homebrew**: A Homebrew formula is published to the `impulia/homebrew-tap` repository (requires a `HOMEBREW_TAP_TOKEN` secret with write access to the tap repo).

## Conventional Commits

To ensure the automated system works correctly, all Pull Request titles must follow the Conventional Commits format:

-   `feat: ...` for new features (triggers a Minor version bump).
-   `fix: ...` for bug fixes (triggers a Patch version bump).
-   `chore:`, `docs:`, `refactor:`, `style:`, `test:`, `ci:` for changes that don't affect the version or trigger a Patch bump depending on configuration.
-   Adding `!` after the type or `BREAKING CHANGE:` in the footer triggers a Major version bump.

This is enforced by the `mergeable` check on all Pull Requests.

## Distribution Channels

-   **GitHub Releases**: Compiled macOS binaries (aarch64 + x86_64) attached to each release.
-   **Self-Update**: `neflo self-update` downloads the latest binary from GitHub Releases.
-   **Homebrew**: `brew install impulia/tap/neflo` (requires the `impulia/homebrew-tap` repository to exist).
-   **Shell Installer**: A shell installer script is generated by `cargo-dist` for each release.

## Manual Steps (Rare)

Under normal circumstances, no manual steps are required for a release. The system is designed to be "hands-off."

If you need to manually trigger a release or fix a failed release, you can use the "Release" workflow's `workflow_dispatch` trigger in the GitHub Actions tab.

### Homebrew Tap Setup (One-Time)

To enable Homebrew distribution:

1.  Create a public repository at `github.com/impulia/homebrew-tap`.
2.  Add a `HOMEBREW_TAP_TOKEN` secret to the `neuroflow` repository — a GitHub PAT with `repo` scope for the tap repository.
